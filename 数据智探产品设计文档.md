### 1.2 版本边界

V1 包含：

- 公共资源、个人空间和团队空间选择。
- 数据资产概览。
- 数据库、表、字段数据字典浏览。
- 自然语言生成 SQL。
- SQL 安全校验与只读执行。
- 查询结果表格、图表、AI 结论、SQL、执行过程和数据来源。
- 基于同一查询会话的连续追问。
- 查询历史。
- 权限控制、脱敏、审计及异常处理。

V1 不包含：

- 数据写入、修改、删除及数据库 DDL。
- 用户通过自然语言直接执行交易。
- 任意 Python 代码执行。
- 跨租户数据查询。
- 超大数据集离线计算任务编排。
- 用户自定义数据清洗和 ETL。

### 1.3 V1 默认实施决策

为避免研发因待确认事项停滞，评审未另行决议时，V1 按以下默认值实施：

| 项目 | V1 默认决策 |
| --- | --- |
| 查询模式 | 异步任务；创建查询后通过 SSE 接收状态，最终通过结果接口取数 |
| 查询范围 | 单次查询允许关联同一查询引擎内的多表；默认不做跨数据源联邦查询 |
| 数据库权限 | 使用只读账号，并由 SQL 网关做第二层对象权限、行级权限和成本校验 |
| SQL 开放范围 | 普通用户仅查看执行过程；具备 `query:sql:read` 才能查看 SQL，具备 `query:sql:execute` 才能编辑执行 |
| 结果展示 | 首屏 100 行，服务端分页；在线查询最多返回 10,000 行 |
| 导出 | 10,000 行以内同步 CSV；超过上限拒绝并提示缩小范围，V1 不做异步大文件导出 |
| 查询保存 | 查询元数据及审计记录按公司策略保存；产品侧默认展示最近 90 天 |
| 连续追问 | 通过 `sessionId + parentQueryId` 建立父子关系，每轮生成独立 `queryId` |
| 模型输出 | LLM 只生成结构化意图、候选 SQL、解释和图表建议，最终权限与执行权始终在后端 |
| 时间格式 | 接口统一使用带时区的 ISO 8601；数据库存储 UTC，前端按用户时区展示 |

本表属于 V1 基线。若第 15 章事项在评审中产生不同结论，应先更新本文档再进入联调。

## 2. 产品背景与目标

### 2.1 用户问题

当前数据库使用存在以下障碍：

1. 用户不知道数据库中有哪些可用数据。
2. 用户不了解表、字段及表间关系。
3. 业务人员不会编写 SQL。
4. 专业用户编写 SQL 前需要反复查数据字典。
5. 查询结果缺少业务解释、图表和数据来源说明。
6. 数据权限、安全和查询资源消耗难以统一控制。

### 2.2 产品目标

数据智探应同时成为：

- 投研数据库的数据资产门户。
- 面向业务用户的自然语言查询工具。
- 面向专业用户的可解释 NL2SQL 工作台。

核心价值链：

```text
发现数据 → 描述问题 → 生成并验证 SQL → 执行查询
        → 解释结果 → 图表展示 → 连续追问 → 复用查询
```

### 2.3 成功指标

| 指标 | V1 目标 |
| --- | --- |
| 自然语言查询成功率 | ≥ 85% |
| 首次生成 SQL 可执行率 | ≥ 90% |
| 查询结果业务正确率 | 抽样评审 ≥ 85% |
| 数据来源可追溯率 | 100% |
| 越权查询拦截率 | 100% |
| 普通查询 P95 完成时间 | ≤ 15 秒 |
| 用户主动修改 SQL 比例 | 用于观测，不设硬目标 |
| 查询后继续追问比例 | 用于衡量产品价值 |

## 3. 用户角色与权限

### 3.1 用户角色

| 角色 | 典型需求 |
| --- | --- |
| 投研人员 | 查询行情、财务、行业、舆情和组合数据 |
| 量化研究员 | 查看 SQL、修改 SQL、导出结果、复用查询 |
| 组合经理 | 查询组合收益、风险暴露和持仓变化 |
| 数据管理员 | 维护数据字典、别名、关联关系和质量信息 |
| 系统管理员 | 配置数据源、权限、限流和审计策略 |

### 3.2 权限原则

权限控制必须发生在以下三个阶段：

1. 数据字典检索前：用户不可检索无权限的数据库、表和字段。
2. SQL 生成后：校验 SQL 中引用的所有对象。
3. SQL 执行前：数据库账户或网关再次进行权限校验。

权限粒度：

- 数据源权限。
- Database/Schema 权限。
- 表权限。
- 字段权限。
- 行级权限。
- 导出权限。
- 查看 SQL 权限。
- 编辑并执行 SQL 权限。

敏感字段应根据规则进行隐藏或脱敏，LLM 不得看到用户无权访问的原始值。

## 5. 页面与交互设计

### 5.1 页面头部

页面标题为“数据智探”，副标题为“数据资产浏览 · 自然语言查询 · 自动分析与可视化”。

功能要求：

- 返回按钮回到 AlphaOctopus 首页。
- 数据状态展示正常、延迟、异常三种状态。
- “浏览数据资产”打开数据字典抽屉。
- “查询历史”打开历史记录抽屉或页面。
- 页面不向普通用户暴露底层数据库或查询引擎选择。
- 系统根据资源空间、数据权限和问题涉及的数据自动选择底层数据源。
- 如需人工指定查询引擎，应作为专业用户的高级功能单独设计，不纳入 V1。

### 5.2 数据资产概览

数据资产首先按资源归属分为三个空间：

| 空间 | 定义 | 默认权限 |
| --- | --- | --- |
| 公共资源 | 平台统一采购、生产和维护的标准数据 | 按平台授权范围只读 |
| 个人空间 | 用户上传、加工、保存或单独授权的数据 | 仅数据所有者可见 |
| 团队空间 | 团队成员共同维护和共享的研究数据 | 按团队成员及角色授权 |

页面默认选中公共资源。切换资源空间后，统计指标、资源分类和数据字典检索范围必须同步变化。切换到团队空间时，若用户属于多个团队，应展示团队选择器。

展示当前资源空间的：

- 数据库数量。
- 数据表数量。
- 字段数量。
- 数据质量评分。
- 最近成功同步时间。

空间卡片展示：

- 空间名称和图标。
- 空间说明。
- 数据库及数据表数量。
- 权限或维护主体，例如“平台维护”“仅自己可见”“量化研究组”。
- 当前选中状态。

业务域卡片展示：

- 业务域名称和图标。
- 数据表数量。
- 简介。
- 数据更新时间或异常状态。

点击业务域卡片打开数据资产抽屉，并自动应用对应业务域筛选条件。

#### 5.2.1 指标统计范围

所有概览指标必须基于“当前用户在当前资源空间下实际可见的数据资产集合”计算，不得先统计全量资产再仅在前端隐藏。

定义可见资产集合 `A`：

```text
A = 满足以下全部条件的数据资产
1. resource_space_id 等于当前选择的公共、个人或团队空间
2. 团队空间的 team_id 等于当前选择的团队
3. 资产未被逻辑删除
4. 资产状态不是 hidden 或 disabled
5. 当前用户至少具有 metadata:read 权限
6. 不属于系统内部 Schema，例如 information_schema
```

纳入统计的资产类型：

- 物理表 `TABLE`。
- 视图 `VIEW`。
- 物化视图 `MATERIALIZED_VIEW`。

不纳入统计：

- 临时表。
- 系统表。
- 已删除或已禁用资产。
- 用户无元数据查看权限的资产。
- 查询执行过程中产生且未保存的临时结果。

个人保存的查询结果只有在用户明确执行“保存为数据集”后，才计入个人空间数据表数量。

#### 5.2.2 数据库、数据表和字段数量

##### 数据库数量

数据库唯一键：

```text
connection_id + catalog_name + database_name
```

计算方式：

```sql
COUNT(
  DISTINCT connection_id,
           catalog_name,
           database_name
)
```

规则：

- 同一个数据库通过两个不同连接接入，按两个数据库统计。
- 同一连接中的不同 Schema 是否作为数据库统计，由数据源适配器统一映射。
- MySQL 的 Database、PostgreSQL 的 Database + Schema、Hive 的 Database 等差异必须在适配层归一化。
- 页面统一称为“库”，不直接暴露底层方言差异。

##### 数据表数量

```text
tableCount = 可见资产集合 A 中 TABLE、VIEW、MATERIALIZED_VIEW 的数量
```

同一物理表被挂载到不同资源空间时，在各自空间分别计数；在同一空间内不得重复计数。

##### 字段数量

```text
fieldCount = Σ 每张可见表中当前用户有 metadata:read 权限的字段数
```

规则：

- 无权限字段不计入。
- 已废弃但仍可查询的字段计入，并在字典中标记 `deprecated`。
- 计算字段和虚拟字段计入，但需标记字段类型。
- 同一字段在不同表出现时分别计数。

##### 后端伪代码

```ts
const visibleAssets = await assetRepository.findVisibleAssets({
  userId,
  resourceSpaceId,
  teamId,
  statuses: ["active", "deprecated"],
  assetTypes: ["TABLE", "VIEW", "MATERIALIZED_VIEW"],
});

const databaseCount = distinctBy(
  visibleAssets,
  (asset) =>
    `${asset.connectionId}:${asset.catalogName}:${asset.databaseName}`,
).length;

const tableCount = visibleAssets.length;

const fieldCount = await fieldRepository.countVisibleFields({
  userId,
  assetIds: visibleAssets.map((asset) => asset.id),
});
```

#### 5.2.3 数据质量评分

数据质量不是人工填写的百分比，而是最近一次有效质量检测任务的聚合结果。

V1 使用五个质量维度：

| 维度 | 含义 | 默认权重 |
| --- | --- | ---: |
| 完整性 Completeness | 必填字段是否缺失 | 25% |
| 新鲜度 Freshness | 数据是否按 SLA 更新 | 30% |
| 有效性 Validity | 类型、范围、枚举、格式是否符合规则 | 20% |
| 唯一性 Uniqueness | 主键或业务唯一键是否重复 | 15% |
| 一致性 Consistency | 跨字段、跨表关系是否一致 | 10% |

权重由系统配置统一管理。V1 不允许每个用户自行修改，以保证不同空间的评分可比较。

##### 单条质量规则评分

完整性、有效性、唯一性规则使用统一公式：

```text
ruleScore =
  checkedRowCount = 0
    ? null
    : max(
        0,
        100 × (
          1 - violationRowCount / checkedRowCount
        )
      )
```

示例：

```text
检查 10,000 行
发现 revenue 为空 35 行

完整性得分
= 100 × (1 - 35 / 10,000)
= 99.65
```

一致性规则可以返回违反行数，也可以返回通过/失败：

```text
有行级结果：
使用统一 ruleScore 公式

只有通过/失败：
通过 = 100
失败 = 0
```

##### 新鲜度评分

每张需要同步的数据表必须配置：

```ts
interface FreshnessSla {
  expectedAvailableAt: string;
  expectedIntervalSeconds: number;
}
```

其中：

- `expectedAvailableAt`：当前周期数据应当可用的最晚时间。
- `expectedIntervalSeconds`：正常更新间隔，例如日表为 86,400 秒，分钟表为 60 秒。

计算：

```text
overdueSeconds =
  max(0, serverNow - expectedAvailableAt)

decayWindow =
  3 × expectedIntervalSeconds

freshnessScore =
  max(
    0,
    100 × (
      1 - overdueSeconds / decayWindow
    )
  )
```

说明：

- 未超过应到时间，得分为 100。
- 超期一个正常更新周期，约为 66.67。
- 超期两个正常更新周期，约为 33.33。
- 超期三个及以上更新周期，得分为 0。
- 使用服务端时间计算，不能使用浏览器本地时间。
- 没有配置 SLA 的表，新鲜度得分为 `null`，不得默认为 100。

示例：

```text
日表预期最晚 08:00 可用
当前服务端时间为次日 08:00
超期 24 小时
正常间隔 24 小时

新鲜度
= 100 × (1 - 24 / 72)
= 66.67
```

##### 单表维度得分

一个维度可能配置多条规则。维度得分按规则重要度加权：

```text
dimensionScore =
  Σ(ruleScore × ruleWeight)
  / Σ(ruleWeight)
```

只计算本次成功执行且有检查数据的规则。执行失败的规则不直接记 0，而是降低质量覆盖率并产生质量任务异常。

##### 单表综合质量得分

```text
tableQualityScore =
  Σ(dimensionScore × dimensionWeight)
  / Σ(有有效得分的 dimensionWeight)
```

同时计算质量覆盖率：

```text
tableQualityCoverage =
  Σ(有有效得分的 dimensionWeight)
  / Σ(全部 dimensionWeight)
```

示例：

```text
完整性 98，权重 25
新鲜度 100，权重 30
有效性 95，权重 20
唯一性 100，权重 15
一致性未配置，权重 10

tableQualityScore
= (98×25 + 100×30 + 95×20 + 100×15)
  / (25+30+20+15)
= 98.33

tableQualityCoverage
= 90 / 100
= 90%
```

##### 资源空间综合质量得分

每张表配置重要度：

| 重要度 | 权重 |
| --- | ---: |
| core 核心表 | 3 |
| important 重要表 | 2 |
| normal 普通表 | 1 |

空间质量得分：

```text
spaceQualityScore =
  Σ(tableQualityScore × tableImportanceWeight)
  / Σ(tableImportanceWeight)
```

只有 `tableQualityCoverage ≥ 60%` 且最近质量任务仍在有效期内的表参与分子。

空间质量覆盖率：

```text
spaceQualityCoverage =
  Σ(有有效质量得分的表权重)
  / Σ(当前可见表权重)
```

当 `spaceQualityCoverage < 80%` 时：

- 接口仍返回内部计算分数。
- `quality.status` 返回 `insufficient_coverage`。
- 前端不展示误导性的百分比，显示“质量规则待完善”。

质量任务有效期：

```text
qualityValidUntil =
  min(
    qualityRun.finishedAt
      + max(24小时, expectedIntervalSeconds × 2),
    下一次计划质量任务应完成时间
  )
```

超过有效期的检测结果不得作为当前质量评分。

##### 前端展示规则

分数保留一位小数：

```ts
displayScore =
  Math.round(score * 10) / 10;
```

状态和颜色：

| 得分/状态 | 文案 | 颜色 |
| --- | --- | --- |
| 95–100 | 优秀 | 绿色 |
| 85–94.9 | 良好 | 蓝色 |
| 70–84.9 | 待关注 | 橙色 |
| 0–69.9 | 异常 | 红色 |
| 覆盖率不足 | 质量规则待完善 | 灰色 |
| 质量任务失败 | 质量检测异常 | 红色 |

页面当前使用百分号展示，例如 `96.8%`。这里的百分号表示 100 分制质量评分，不表示“有 96.8% 的表正确”。Tooltip 必须展示：

```text
综合质量：96.8
质量覆盖率：92.4%
完整性：98.1
新鲜度：97.5
有效性：96.2
唯一性：99.0
一致性：91.3
检测时间：2026-07-03 09:30
```

#### 5.2.4 最近成功同步

页面文案应使用“最近成功同步”，避免“最近同步”无法区分成功或失败。

##### 表级时间

```text
table.lastSuccessfulSyncAt =
  当前数据版本最近一次状态为 SUCCESS 的同步任务完成时间
```

规则：

- 使用同步任务的服务端 `finished_at`。
- 失败、取消、运行中的任务不更新该字段。
- 全量和增量同步都可以更新。
- 手工上传成功后，上传完成时间视为该个人数据集的成功同步时间。
- 虚拟视图没有独立同步任务时，继承其上游依赖中最旧的成功可用时间，用于反映短板。

##### 资源空间时间

```text
space.lastSuccessfulSyncAt =
  max(
    当前可见表的 table.lastSuccessfulSyncAt
  )
```

这个指标表示“空间内最近发生的一次成功同步”，不代表所有表都已更新。因此页面必须同时展示整体同步状态，不能仅凭这个时间判断空间健康。

##### 业务域时间

```text
domain.lastSuccessfulSyncAt =
  max(
    该业务域可见表的
    table.lastSuccessfulSyncAt
  )
```

##### 相对时间格式

接口同时返回 `serverTime`。前端使用：

```text
elapsedSeconds =
  max(
    0,
    serverTime - lastSuccessfulSyncAt
  )
```

显示规则：

| 时间差 | 显示 |
| --- | --- |
| `< 60秒` | 刚刚 |
| `1–59分钟` | `N 分钟前`，向下取整 |
| `1–23小时` | `N 小时前`，向下取整 |
| `1–6天` | `N 天前`，向下取整 |
| `≥ 7天` | `YYYY-MM-DD HH:mm` |
| 无成功记录 | `暂无成功同步` |

鼠标悬停始终显示绝对时间和时区：

```text
2026-07-03 09:52:16 Asia/Shanghai
```

#### 5.2.5 整体同步状态

最近成功同步时间不能表示是否存在失败或延迟。整体状态基于当前可见表的重要度加权计算。

每张表同步状态：

| 状态 | 判定 |
| --- | --- |
| `healthy` | 未超过 `expectedAvailableAt`，最近任务成功 |
| `syncing` | 当前有运行任务，且未延迟 |
| `delayed` | 已超过 `expectedAvailableAt` |
| `failed` | 当前周期最近一次任务失败 |
| `unknown` | 未配置同步任务或 SLA |

定义：

```text
failedWeightRatio =
  失败表权重 / 已配置同步的表权重

delayedWeightRatio =
  延迟表权重 / 已配置同步的表权重
```

空间状态优先级：

```text
1. error
   任意核心表 failed
   或 failedWeightRatio ≥ 10%

2. delayed
   任意核心表 delayed
   或 delayedWeightRatio ≥ 5%

3. syncing
   存在运行中任务，且没有 error/delayed

4. healthy
   其余已配置且正常的情况

5. unknown
   没有任何同步配置
```

前端展示：

| 状态 | 文案 | 颜色 |
| --- | --- | --- |
| `healthy` | 数据更新正常 | 绿色 |
| `syncing` | 数据同步中 | 蓝色 |
| `delayed` | 部分数据延迟 | 橙色 |
| `error` | 数据同步异常 | 红色 |
| `unknown` | 同步状态未知 | 灰色 |

点击状态标签打开同步详情，至少展示：

- 失败表数量。
- 延迟表数量。
- 核心异常表。
- 最近失败原因。
- 预计下一次更新时间。

#### 5.2.8 概览指标计算示例

假设量化研究团队空间有 4 张可见表：

| 表 | 重要度权重 | 表质量分 | 覆盖率 | 同步状态 | 最近成功同步 |
| --- | ---: | ---: | ---: | --- | --- |
| daily_quote | 3 | 99 | 100% | healthy | 09:52 |
| financial_indicator | 3 | 96 | 90% | delayed | 前日 22:00 |
| factor_value | 2 | 92 | 80% | healthy | 09:40 |
| temp_result | 1 | 无 | 0% | unknown | 无 |

空间质量：

```text
spaceQualityScore
= (99×3 + 96×3 + 92×2)
  / (3+3+2)
= 96.125
≈ 96.1
```

质量覆盖率：

```text
(3+3+2) / (3+3+2+1)
= 88.89%
```

因此前端显示：

```text
数据质量 96.1%
质量覆盖率 88.9%
```

同步状态：

```text
financial_indicator 是核心表且 delayed
因此空间同步状态 = delayed
```

最近成功同步：

```text
max(09:52, 前日22:00, 09:40, null)
= 09:52
```

因此页面显示：

```text
部分数据延迟
最近成功同步：8 分钟前
```

两者必须同时出现，不能因为“8 分钟前有一张表成功”就显示“数据更新正常”。

### 5.3 自然语言查询

输入要求：

- 支持中文、英文及中英文混合。
- 最大长度 2,000 字符。
- Enter 执行，Shift+Enter 换行。
- 空内容不可执行。
- 执行过程中按钮变为“停止”。
- 用户可点击推荐问题填入输入框。

查询前端状态：

| 状态 | 说明 | 页面表现 |
| --- | --- | --- |
| idle | 未执行 | 显示查询按钮 |
| understanding | 意图识别 | 展示“正在理解问题” |
| retrieving | 检索数据字典 | 展示匹配表和字段数量 |
| generating | 生成 SQL | 展示“正在生成查询” |
| validating | 安全和成本校验 | 展示校验状态 |
| executing | 数据库执行 | 展示耗时，可停止 |
| explaining | 生成结论和图表 | 结果区域骨架屏 |
| success | 完成 | 展示完整结果 |
| failed | 失败 | 展示错误原因和可恢复操作 |
| cancelled | 用户取消 | 保留输入，可重新执行 |

状态流转必须由后端任务状态驱动，前端不得根据计时器自行推断：

```text
idle
  → understanding
  → retrieving
  → clarification_required ──用户回答──→ retrieving
  → generating
  → validating
  → executing
  → explaining
  → success
```

终止规则：

- `understanding` 至 `explaining` 任一运行态均可进入 `failed`。
- `understanding` 至 `explaining` 接受取消后进入 `cancelling`，后端任务真正终止后进入 `cancelled`。
- `success`、`failed`、`cancelled` 是终态，不得再回到运行态。
- 重新生成、修改问题、重跑历史查询均创建新的 `queryId`，不得覆盖原任务。
- 同一事件序号只能处理一次；迟到事件的序号小于等于当前序号时，前端必须丢弃。

### 5.4 歧义澄清

以下情况不得直接猜测，应向用户澄清：

- 指标存在多个口径，例如“利润”可能指净利润或归母净利润。
- 时间范围缺失且会显著影响结果。
- “最大”“最好”等排序条件没有明确指标。
- 同名业务实体无法唯一匹配。
- 查询涉及多个无法自动判断的业务域。

澄清形式：

```text
你提到的“净利润”希望使用哪种口径？
[归母净利润] [扣非归母净利润] [净利润]
```

用户选择后继续原查询，不新建查询会话。

### 5.5 查询结果

#### 5.5.1 AI 结论

应包含：

- 直接回答用户问题。
- 结果数量和关键分布。
- 重要异常或限制说明。
- 统计口径及时间范围。

不得包含查询结果无法支持的推断。若结果为空，应解释可能原因并建议调整条件。

#### 5.5.2 数据表

- 默认最多展示 100 行。
- 支持分页、排序、列宽调整和固定列。
- 数值按字段单位格式化。
- 股票代码、日期等字段使用统一格式。
- 支持导出 CSV；导出受权限和最大行数限制。
- 服务端返回结果超过限制时提示用户缩小范围或创建离线任务。

#### 5.5.3 图表

自动图表规则：

| 数据形态 | 默认图表 |
| --- | --- |
| 时间 + 数值 | 折线图 |
| 分类 + 数值 | 柱状图 |
| 占比数据 | 饼图 |
| 两个连续变量 | 散点图 |
| 股票 + 涨跌 | 热力图 |
| 不适合可视化 | 不自动生成 |

用户可切换图表类型、维度和指标。切换只改变展示，不重新执行 SQL。

#### 5.5.4 SQL

- 默认折叠。
- 有权限用户可复制 SQL。
- 具备编辑权限的用户可修改并重新执行。
- 修改后的 SQL 标记为“用户编辑”，与 AI 原始 SQL 分开审计。
- SQL 格式化展示，并高亮表名和字段名。

#### 5.5.5 执行过程

展示：

- 识别的实体、指标、时间和筛选条件。
- 检索到的数据字典范围。
- SQL 生成与校验结果。
- 数据库执行耗时。
- 结果解释和图表生成状态。

不得展示模型思维链或内部敏感提示词。

#### 5.5.6 数据来源

每次成功查询必须展示：

- 数据源、数据库、Schema、表。
- 实际使用的字段。
- 数据更新时间。
- 指标口径说明。
- 表间关联关系。

### 5.6 连续追问

### 5.9 页面通用状态

每个独立数据区域必须实现以下状态，禁止用演示数字或上一次空间的数据代替失败结果：

| 状态 | 展示要求 | 可用操作 |
| --- | --- | --- |
| 首次加载 | 与最终布局一致的骨架屏 | 无 |

- 新空间接口失败时不得继续展示上一空间的概览数字。

## 7. NL2SQL 核心流程

```text
1. 输入预处理
2. 权限范围计算
3. 意图、实体、指标、时间识别
4. 数据字典及语义指标检索
5. 歧义检测
6. SQL 生成
7. SQL AST 解析
8. 权限、安全、语法、成本校验
9. 必要时自动修复，最多两次
10. 只读执行
11. 结果质量检查
12. AI 结论及图表配置生成
13. 保存审计记录
```

### 7.1 SQL 安全规则

必须满足：

- 仅允许 `SELECT` 和只读 CTE。
- 禁止多语句。
- 禁止注释绕过。
- 禁止 `INSERT`、`UPDATE`、`DELETE`、`MERGE`、DDL、授权命令。
- 禁止调用未授权函数、外部文件和网络函数。
- 必须包含最大行数限制。
- 必须限制执行时间、扫描行数和内存。
- 必须通过 SQL AST 判断，不得仅使用字符串匹配。
- 引用对象必须全部在权限白名单中。

### 7.2 自动修复

以下错误允许自动修复：

- 字段或表别名错误。
- SQL 方言不匹配。
- 明确的类型转换错误。
- 聚合和 GROUP BY 错误。

最多自动修复两次。仍失败时返回可理解错误，不得无限重试。

## 8. 接口设计

以下为建议契约，最终路径可按后端规范调整。

### 8.0 通用接口约定

请求约定：

- 所有接口从登录态获取 `userId`、租户和角色，前端不得传入或覆盖。
- 写接口支持请求头 `Idempotency-Key`，相同用户、相同键在 24 小时内不得创建重复任务。
- 请求头携带 `X-Request-Id`；未传时由网关生成，并在响应头返回。
- 分页统一使用 `page`（从 1 开始）和 `pageSize`（默认 20，最大 200）。
- 列表排序必须使用后端白名单字段，不得直接拼接前端传入的列名。
- 接口版本发生不兼容变更时使用 `/api/v2`，不得静默改变字段语义。

成功响应：

```ts
interface ApiResponse<T> {
  requestId: string;
  data: T;
}

interface PageResult<T> {
  items: T[];
  page: number;
  pageSize: number;
  total: number;
}
```

第 8 章后续 JSON 示例为便于阅读，仅展示 `data` 字段内容；实际 HTTP 响应仍使用上述统一外层结构。

失败响应：

```ts
interface ApiErrorResponse {
  requestId: string;
  error: {
    code: string;
    message: string;
    retryable: boolean;
    details?: Record<string, unknown>;
  };
}
```

HTTP 状态码：

### 8.6 查询事件流

`GET /api/ai-query/queries/{queryId}/events`

建议使用 SSE，事件类型：

- `query.status`
- `query.clarification_required`
- `query.schema_retrieved`
- `query.sql_generated`
- `query.validation_completed`
- `query.execution_progress`
- `query.result_ready`
- `query.failed`
- `query.cancelled`

事件必须包含递增序号，支持断线续传。客户端重连时通过标准请求头 `Last-Event-ID` 传最后成功处理的事件 ID。

统一事件结构：

```ts
interface QueryEvent<T = unknown> {
  eventId: string;
  sequence: number;
  queryId: string;
  type:
    | "query.status"
    | "query.clarification_required"
    | "query.schema_retrieved"
    | "query.sql_generated"
    | "query.validation_completed"
    | "query.execution_progress"
    | "query.result_ready"
    | "query.failed"
    | "query.cancelled";
  occurredAt: string;
  data: T;
}
```

关键事件数据：

```ts
interface QueryStatusEvent {
  status: QueryStatus;
  displayMessage: string;
}

interface ClarificationRequiredEvent {
  clarificationId: string;
  question: string;
  mode: "single" | "multiple" | "text";
  options?: Array<{
    id: string;
    label: string;
    description?: string;
  }>;
  expiresAt: string;
}

interface ExecutionProgressEvent {
  elapsedMs: number;
  scannedRows?: number;
  stageMessage: string;
}

interface QueryFailedEvent {
  code: string;
  message: string;
  retryable: boolean;
}
```

SSE 规则：

- 服务端每 15 秒发送心跳，心跳不增加业务状态序号。
- 事件至少保留 30 分钟，确保短时断线可补发。
- 同一 `queryId` 的 `sequence` 从 1 递增且不可重复。
- `query.sql_generated` 仅向具备 SQL 查看权限的用户发送 SQL；其他用户只发送“已生成”状态。
- 终态事件发送后服务端关闭连接。

### 8.8 取消查询

`POST /api/ai-query/queries/{queryId}/cancel`

接口接受运行态任务。收到请求先返回 `cancelling`；取消成功后必须终止数据库任务及模型生成任务并进入 `cancelled`。终态任务重复取消需幂等返回当前终态。

### 8.9 执行用户编辑 SQL

`POST /api/ai-query/sql/execute`

请求必须包含原 `queryId`、编辑后的 SQL 和原查询的 `schemaVersion`。服务端必须重新进行权限、安全、Schema 版本和成本校验，并创建新的查询任务，标记 `editedByUser=true` 和 `sourceQueryId`。

### 8.10 查询历史

- `GET /api/ai-query/history`
- `DELETE /api/ai-query/history/{queryId}`
- `PUT /api/ai-query/history/{queryId}/favorite`
- `DELETE /api/ai-query/history/{queryId}/favorite`

删除仅移除用户侧历史展示，不删除安全审计记录。重新执行历史查询需调用创建查询接口并传 `sourceQueryId`，生成新的查询任务。

### 8.11 导出查询结果

`POST /api/ai-query/queries/{queryId}/exports`

请求：

```json
{
  "format": "csv",
```

## 9. 错误处理

| 错误码 | 场景 | 用户提示 | 可恢复操作 |
| --- | --- | --- | --- |
| AQ-4001 | 问题为空或过长 | 请输入有效问题 | 修改输入 |
| AQ-4002 | 问题存在歧义 | 需要确认指标口径 | 选择澄清项 |
| AQ-4031 | 无资源空间权限 | 无权访问该资源空间或团队数据 | 切换空间/申请权限 |
| AQ-4032 | 表或字段越权 | 查询包含无权访问的数据 | 修改问题 |
| AQ-4101 | 查询结果已过期 | 结果已过期，请重新执行 | 重新执行 |
| AQ-4091 | Schema 已变更 | 数据结构已更新 | 重新生成 |
| AQ-4221 | 无法生成有效 SQL | 暂时无法理解该问题 | 改写/查看示例 |
| AQ-4222 | SQL 安全检查失败 | 查询不符合只读安全规则 | 重新生成 |
| AQ-4291 | 查询过于频繁 | 请求较多，请稍后重试 | 等待 |
| AQ-5001 | 数据库执行失败 | 查询执行失败 | 重试/查看详情 |
| AQ-5041 | 查询超时 | 查询范围过大 | 缩小时间或数据范围 |

错误详情中不得暴露数据库密码、连接地址、内部堆栈或系统提示词。

## 10. 非功能要求

### 10.1 性能

- 页面首屏 P95 ≤ 3 秒。
- 数据字典搜索 P95 ≤ 2 秒。
- 普通查询 P95 ≤ 15 秒。
- 查询状态反馈应在提交后 1 秒内出现。
- 表格前端单页最多渲染 200 行。
- 导出超过 10,000 行应转为后端异步任务。

### 10.2 可用性

- 查询任务支持取消。
- SSE 断开后自动重连，并通过事件序号恢复。
- 数据库暂时不可用时展示状态，不影响浏览历史结果。
- 查询结果至少保存 7 天，具体按公司数据策略配置。

### 10.3 兼容性

- Chrome 最近两个主版本。
- Edge 最近两个主版本。
- 最低页面宽度 1280px；更窄时允许业务域卡片换行。
- 支持中文和英文数据库内容。
- 时间统一存储为 UTC，页面按用户时区展示。

### 10.4 安全与合规

- 所有查询必须记录用户、时间、问题、SQL、数据对象、耗时和状态。
- 查询日志中的敏感值需脱敏。
- LLM 服务不得用于训练企业数据，除非另有明确协议。
- 数据库凭证仅保存在服务端密钥系统。
- 防止数据字典描述或示例值中的提示注入影响系统指令。
- 模型输出不能绕过 SQL AST 校验。

### 13.5 算法正确性评测

建立不少于 300 条标准问题集，覆盖：

- 单表查询。
- 多表关联。
- 时间序列。
- 同比、环比、排名和分组。
- 财务指标口径。
- 组合指标。
- 空值和异常值。
- 歧义问题。
- 无法回答的问题。

每条标准问题应包含：

- 用户问题。
- 允许使用的表和字段。
- 标准 SQL 或等价规则。
- 标准结果摘要。
- 是否应澄清。
- 是否应拒绝。

评测维度：

- Schema Linking 准确率。
- SQL 执行正确率。
- 结果一致率。
- 应澄清召回率。
- 应拒绝召回率。
- 结论忠实度。

## 16. 开发就绪检查表

### 16.1 联调前必须提供

- 经评审的页面原型和各区域 loading、empty、error、no-permission 状态。
- OpenAPI 3.x 文件，字段、枚举、必填性和错误码与第 8 章一致。
- 至少一个公共空间、一个个人空间、两个团队空间的测试账号及权限矩阵。
- 可用于联调的数据字典样本，至少包含 20 张表、跨表关系、敏感字段和中文别名。
- SSE 模拟服务或事件样例，覆盖成功、澄清、失败、取消和断线续传。
- SQL 安全测试库，不得直接使用生产数据库联调。
- 首批不少于 50 条冒烟问题；上线前扩展到第 13.5 节要求的 300 条。

### 16.2 前端完成定义

- 页面与接口字段全部使用共享类型，不自行定义同名枚举。
- 资源空间切换、请求竞态、SSE 重连和事件去重通过自动化测试。
- 结果表格、图表、SQL、来源、执行过程具备完整空态和异常态。
- 无权限用户无法通过 DOM、前端缓存或请求参数看到受限元数据。
- 所有操作错误均展示可追踪的 `requestId`。

### 16.3 后端完成定义

- 接口契约、数据库变更脚本和权限策略通过评审。
- 查询创建具备幂等性；取消能真正终止数据库任务和模型任务。
- SQL 通过 AST、权限、行级、脱敏、成本和只读校验。
- 查询、导出、字典检索和审计日志均按用户及租户隔离。
- 所有异步状态可恢复，服务重启后任务不会永久停留在运行态。
- 关键指标、结构化日志、告警和链路追踪已接入。

### 16.4 测试准入

- P0 待确认事项已有结论。
- OpenAPI 与测试环境实现一致。
- 功能、权限、错误码和埋点均有可验证入口。
- 已准备固定数据集及预期 SQL/结果，避免测试环境数据变化导致无法判定。
- 算法评测结果区分“SQL 可执行”“结果正确”“结论忠实”，不得合并成单一成功率。
